1. **Modular architecture.** Components separate cleanly into modules by responsibility — each understandable, changeable, and testable in isolation, without loading the rest of the system into your head. This keeps us close to microservices patterns even in a monolith: parts improve independently. Modules may nest into sub-modules, and the same test applies recursively — a sub-module is as cohesive and isolated as a top-level one. Modularize for two reasons — to isolate a distinct concern, or to kill duplication. Guard against over-modularizing: extract a *utility* module only once logic repeats in 3+ places; extract a *boundary* module only when it's a genuinely distinct responsibility. The test is isolation, not reuse — reuse is a bonus, not the bar.
2. **High separation of concerns.** The code's dependency graph should be a few highly-cohesive components talking through narrow, stable interfaces — not one tangled web of scripts reaching into each other. The interfaces are the "bridges," and a module is only as good as its interface (narrow, stable, hides internals). Checkable signs: the graph is acyclic, and dependencies point one direction toward stable abstractions. Goal: a clean mental model you can hold in your head.
3. **Code brevity.** We don't chase lines of code — we chase the opposite. Every line must earn its place. We appreciate dense logic, not prose. We don't write just-in-case coverage for scenarios that won't occur in real life; we'd rather fix a real bug in production than ship verbose code guarding against the imaginary. *(Guardrail: this targets unlikely failures, not real ones — known failure modes still get handled.)*
4. **Comments are one-liners.** Every comment earns its place by being brief — and by saying what the code can't say itself (the *why*), not restating what the code already shows.
5. **Reuse open source projects.** For commodity capabilities, if a good open-source project fits, imports light, and shows signs of active maintenance (recent commits, releases, responsive issues), we use it instead of building our own. This lets us keep the code lean. Two limits: don't pull a heavy dependency for what a few lines would solve, and don't outsource capabilities that are *core* to the project.
